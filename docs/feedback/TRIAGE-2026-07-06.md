## Clusters

### T1 — Windows locale-default text decode crashes the whole run (ATTACK; CRITICAL; 1 report, BLOCKER-exempt)

`2026-07-06-tu-v16-overnight-p4-fleet §F-CV-1`. A `UnicodeDecodeError: 'charmap'`
killed run `…T062215Z` starting PR4/6 after 3 green PRs; `PYTHONUTF8=1` on the relaunch
suppressed it over 3 further PRs. **Verified open in source, and distinct from the
shipped 0.1.1 hang fix** — that fix added `stdin=DEVNULL` + hermetic git flags, not
encodings. The decode-unsafe paths are exactly the ones the 0.1.1 pass did *not* touch:

- `interface/proc.py:92` `run_with_timeout` — `text=True`, **no `encoding=`** → cp1252 on
  win32. This is the dominant path: it decodes every gate-check command's stdout/stderr,
  and agent-touched files (emoji, smart quotes, box-drawing) routinely carry bytes ≥0x80.
- `interface/git.py:32` `_run` — `capture_output=True, text=True`, no `encoding=`.
- `interface/drivers/headless.py:230` — `pr.prompt).read_text()` (a non-ASCII prompt crashes).
- `interface/cli.py:49` — `series_file.read_text()`.

By contrast `headless_spawn.py`, `mcp/server.py`, `scaffold.py`, `telemetry_writer.py`
*all* already pass `encoding='utf-8'` — so the fix is inconsistent, and the four unguarded
sites are the leak. Four call sites with one cause ⇒ attack the cause, not each site.

| # | proposed promotion | home | status |
|---|--------------------|------|--------|
| T1a | Give every text-decoding subprocess/`read_text` an explicit `encoding='utf-8', errors='replace'` — `proc.py::run_with_timeout` and `Git._run` (both `Popen`/`subprocess.run`), plus `read_text(encoding='utf-8')` in `headless.py:230` and `cli.py:49`. Prefer a single shared decode helper (or reuse `proc.GIT_HERMETIC_FLAGS`-style module constant) so a new subprocess site can't silently regress. | `interface/proc.py`, `interface/git.py`, `interface/drivers/headless.py`, `interface/cli.py` | proposed |
| T1b | Regression test that would have caught it: a gate check whose command emits a byte ≥0x80 (e.g. `printf '\x90'`) and a prompt file with a non-ASCII char, asserting the run does not raise `UnicodeDecodeError` on win32. The existing stub fixtures are pure-ASCII, so nothing exercised the decode boundary. | `tests/` | proposed |
| T1c | Belt-and-braces: self-enable UTF-8 mode at the CLI/MCP entry (equiv. of `PYTHONUTF8=1`) so a missed future site degrades to replacement rather than a crash. Secondary to T1a — do not let it substitute for fixing the sites. | `interface/cli.py` / `interface/mcp/__main__.py` | proposed |

### T4 — convoy synthesizes a bare `pr.id` commit message (ATTACK; MAJOR; reframed — see ledger)

Reports framed this as "last-PR-of-series ships a bare commit message" — `2026-07-06
§F-CV-3` (2 occurrences: w3b-PR5, W4b-PR4), and the invocation supplies director evidence
of **seven** occurrences across first/middle/last positions (w3b-PR5, W4b-PR4, WS-C-W3
PR-1a/1b/2, WS-C-W6 PR-1, WS-C-W5 PR-1). **Source resolves the ambiguity the reports
left open:** `headless.py:262` calls `git.commit_all(pr.id)` and `git.py:64-77` runs
`git commit -m <message>` with `message == pr.id`. So **every convoy commit is the bare
`pr.id` string** — the engine *never* reads or preserves an agent-authored commit message.
The "middle PRs never did this / last-PR context differs" hypothesis is a red herring
(a director-side artifact of which commits happened to be inspected). This is systemic and
positional-independent, exactly as the invocation states, and the fix is not a warn-gate
bolted on — it is to stop discarding the agent's message.

| # | proposed promotion | home | status |
|---|--------------------|------|--------|
| T4a | Have the implementation agent author the commit (its prompt already carries the conventional-commit rule), and have the driver **preserve** that commit rather than overwriting it with `pr.id`. Concretely: stop the unconditional `git.commit_all(pr.id)` from being the message of record — either let the agent commit and have convoy commit only residual unstaged changes, or pass the agent-authored subject through. Removes the mechanism (`pr.id` as message) instead of papering a check over it. | `interface/drivers/headless.py:262`, `interface/git.py::commit_all` | proposed |
| T4b | Record commit provenance in telemetry — whether the recorded commit was agent-authored or engine-synthesized — so this class is observable rather than director-spotted. Pairs with T4a's verification. **(consumer-affecting if it adds a telemetry field.)** | `core/telemetry.py` (`SpawnComplete` or a new commit field) + `interface/drivers/headless.py` | watch |
